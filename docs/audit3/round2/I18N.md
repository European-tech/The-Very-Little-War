#### I18N-R2-001 [CRITICAL] admin/listesujets.php:8 -- ISO-8859-1 charset declaration in admin page

**File:** `/home/guortates/TVLW/The-Very-Little-War/admin/listesujets.php`
**Line:** 8

```php
<meta http-equiv="Content-Type" content="text/html; charset=iso-8859-1" />
```

This is the only file in the entire codebase declaring ISO-8859-1 as its charset. The actual file on disk is UTF-8 encoded. The database connection is set to `utf8`. This creates a three-way mismatch:

- DB stores data as UTF-8
- PHP file is UTF-8 on disk
- Browser interprets output as ISO-8859-1

Any French accented characters fetched from the DB (forum subject titles, author names) will display as mojibake. The `htmlspecialchars()` calls on lines 73-74 specify `'UTF-8'` encoding but the browser is told ISO-8859-1, creating double corruption.

Additionally, `htmlspecialchars($str, ENT_QUOTES, 'UTF-8')` will silently return an empty string if the input is not valid UTF-8 under PHP 8.x, which could cause data loss in the admin view.

**Fix:**
```php
<meta http-equiv="Content-Type" content="text/html; charset=utf-8" />
```

#### I18N-R2-006 [HIGH] includes/basicprivatephp.php:259 -- Email subject line with raw UTF-8 accented characters and no encoding header

**File:** `/home/guortates/TVLW/The-Very-Little-War/includes/basicprivatephp.php`
**Line:** 259

```php
$sujet = "Debut d'une nouvelle partie";
```

The actual string contains the UTF-8 character `\xc3\xa9` for the accent in "Debut" (should be "Debut" but the file contains accented characters elsewhere). More critically, the subject line is passed raw to `mail()` on line 287 without RFC 2047 encoding. Per RFC 2047, non-ASCII characters in email headers must be encoded using either `=?UTF-8?B?...?=` (Base64) or `=?UTF-8?Q?...?=` (Quoted-Printable).

The subject "Debut d'une nouvelle partie" contains the right single quotation mark which is safe ASCII, but the surrounding email headers use UTF-8 characters in the From name "The Very Little War" which is ASCII-safe. However, this is fragile -- if the subject ever gets accented characters (as it should: "Debut" -> "Debut"), it will be corrupted.

**Fix:** Use `mb_encode_mimeheader()` or `=?UTF-8?B?` encoding:
```php
$sujet = "=?UTF-8?B?" . base64_encode("Debut d'une nouvelle partie") . "?=";
```
Or better yet, use PHPMailer which handles all encoding automatically.

#### I18N-R2-011 [MEDIUM] includes/logger.php:29 -- json_encode without JSON_UNESCAPED_UNICODE mangles French log context

**File:** `/home/guortates/TVLW/The-Very-Little-War/includes/logger.php`
**Line:** 29

```php

```
[INFO] [REGISTER] New player registered | {"login":"Rene","email":"rene@example.com"}
```

**Fix:**
```php
$contextStr = !empty($context) ? ' | ' . json_encode($context, JSON_UNESCAPED_UNICODE) : '';
```

#### I18N-R2-012 [MEDIUM] includes/copyright.php:86,91 -- json_encode without JSON_UNESCAPED_UNICODE for JS alerts

**File:** `/home/guortates/TVLW/The-Very-Little-War/includes/copyright.php`

#### I18N-R2-013 [MEDIUM] Multiple admin files -- stripslashes() is obsolete since PHP 5.4

**Files affected:**
- `/home/guortates/TVLW/The-Very-Little-War/admin/listesujets.php` lines 73-74
- `/home/guortates/TVLW/The-Very-Little-War/admin/supprimerreponse.php` lines 53-54
- `/home/guortates/TVLW/The-Very-Little-War/admin/listenews.php` line 89
- `/home/guortates/TVLW/The-Very-Little-War/admin/redigernews.php` lines 29-30
- `/home/guortates/TVLW/The-Very-Little-War/moderation/index.php` lines 229-230
- `/home/guortates/TVLW/The-Very-Little-War/alliance.php` lines 168, 182
- `/home/guortates/TVLW/The-Very-Little-War/includes/menus.php` line 34
- `/home/guortates/TVLW/The-Very-Little-War/index.php` line 53
- `/home/guortates/TVLW/The-Very-Little-War/maintenance.php` line 9
- `/home/guortates/TVLW/The-Very-Little-War/ecriremessage.php` lines 91-92

`magic_quotes_gpc` was removed in PHP 5.4 (2012). The `stripslashes()` calls are now no-ops at best. At worst, they corrupt data that legitimately contains backslashes. For example, if a user writes `C:\Users\test` in a forum post, `stripslashes()` would turn it into `C:Userstest`.

More critically for i18n: some multi-byte UTF-8 sequences contain byte 0x5C (the backslash byte in ASCII). While this is not a problem for well-formed UTF-8, `stripslashes()` operates on bytes not characters, which could theoretically corrupt certain Shift_JIS or other legacy-encoded data if it ever enters the system.

**Fix:** Remove all `stripslashes()` calls. Data from MySQL via `mysqli` is never automatically escaped.

#### I18N-R2-014 [MEDIUM] includes/connexion.php:20 -- DB charset is 'utf8' not 'utf8mb4'

**File:** `/home/guortates/TVLW/The-Very-Little-War/includes/connexion.php`
**Line:** 20

```php
mysqli_set_charset($base, 'utf8');
```

MySQL's `utf8` charset is actually `utf8mb3` -- it only supports characters in the Basic Multilingual Plane (up to U+FFFF). This means:

1. Emoji characters (U+1F600+) will be silently truncated or cause errors
2. Some CJK characters and mathematical symbols are not supported
3. Modern best practice is `utf8mb4` which supports the full Unicode range

Since the game's BBCode supports emoji via PHP (the `bbcode.php` file converts smileys to emoji Unicode characters like U+1F603, U+1F60E, U+1F622), these emoji may cause issues if users try to insert them via other means or if the BBCode-generated emoji are ever stored back to the database.

**Fix:**
```php
mysqli_set_charset($base, 'utf8mb4');
```
Note: This also requires the database tables and columns to use `utf8mb4` collation. Check with:
```sql
ALTER DATABASE tvlw CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
```

#### I18N-R2-019 [MEDIUM] includes/basicpublicphp.php:29 -- mb_strtolower without explicit encoding parameter

**File:** `/home/guortates/TVLW/The-Very-Little-War/includes/basicpublicphp.php`
**Line:** 29

```php
$loginInput = ucfirst(mb_strtolower(trim($_POST['loginConnexion'])));
```

Also in:
- `/home/guortates/TVLW/The-Very-Little-War/inscription.php` line 19
- `/home/guortates/TVLW/The-Very-Little-War/comptetest.php` lines 24, 53

The `mb_strtolower()` calls do not specify an encoding parameter. They rely on `mbstring.internal_encoding` which defaults to UTF-8 in PHP 8.x but was historically `ISO-8859-1` in older PHP versions. While the current server runs PHP 8.2 where the default is correct, explicit encoding is best practice for portability:

```php
$loginInput = ucfirst(mb_strtolower(trim($_POST['loginConnexion']), 'UTF-8'));

Since login validation restricts to `[a-zA-Z0-9_]`, this is safe for now, but if Internationalized usernames are ever supported, `mb_ucfirst()` (PHP 8.4+) or `mb_strtoupper(mb_substr($s, 0, 1, 'UTF-8'), 'UTF-8') . mb_substr($s, 1, null, 'UTF-8')` would be needed.

**Fix:** Add `'UTF-8'` as second parameter to all `mb_strtolower()` calls.

#### I18N-R2-020 [MEDIUM] number_format() uses space as thousands separator -- locale-dependent display

**Files affected:** Multiple (game_actions.php, game_resources.php, guerre.php, moderation/index.php, etc.)

All `number_format()` calls use:
```php
number_format($value, 0, ' ', ' ')
```

This uses a regular space (U+0020) as both decimal and thousands separator. For French locale, the proper thousands separator is a narrow non-breaking space (U+202F) or at minimum a non-breaking space (U+00A0) per French typographic conventions. Using a regular space means:
1. Numbers can line-break mid-digit in responsive layouts
2. Copy-pasting numbers includes invisible spaces that break numeric parsing

This is a cosmetic/i18n standards issue rather than a bug.

**Fix (optional, low priority):** Use `\xC2\xA0` (UTF-8 for non-breaking space) or use `Intl\NumberFormatter` for locale-aware formatting.

**Lines:** 337-357

The BBCode processor converts text emoticons to Unicode emoji:
```php
$text = preg_replace('!:(-)?D!isU', "\xF0\x9F\x98\x83", $text); // U+1F603
$text = preg_replace('!xD!isU', "\xF0\x9F\x98\x86", $text);     // U+1F606
```

These are 4-byte UTF-8 sequences (supplementary plane characters). If BBCode-processed text is ever stored back to the database (e.g., cached or logged), the `utf8` (utf8mb3) charset will silently truncate at the emoji, potentially corrupting subsequent text. This ties to I18N-R2-014.

**Fix:** Ensure database uses `utf8mb4` if BBCode output is stored, or keep emoji conversion output-only.
