## CRITICAL Findings

### [I18N-R1-001] CRITICAL admin/listesujets.php:8 -- ISO-8859-1 charset declaration instead of UTF-8

```html
<meta http-equiv="Content-Type" content="text/html; charset=iso-8859-1" />
```

The admin page declares `charset=iso-8859-1` while the rest of the application uses UTF-8. This means French accented characters (e, a, c) stored as UTF-8 in the database will display as mojibake (garbled characters) on this page. All other pages use `charset=utf-8` via `includes/meta.php`.

**Fix:** Replace with `<meta charset="utf-8">` or include `meta.php`.

---

### [I18N-R1-002] CRITICAL includes/menus.php:30 -- Corrupted double-encoded character in date format string

```php
echo date('d/m/Y Ã H\hi', $donnees['timestamp']);
```

The character `Ã` is the result of double-encoding the UTF-8 `a` character. The intended string is `d/m/Y a H\hi` (displaying as "01/01/2026 a 14h30"). This corrupted character is rendered to every user viewing the news section on the public homepage.

**Fix:** Replace `Ã` with `a` in the date format string: `date('d/m/Y \a H\hi', $donnees['timestamp'])` (escaped `a` to prevent it being interpreted as a date format character, or use the literal `a` if PHP date() does not assign meaning to lowercase `a` -- note: lowercase `a` in date() means am/pm, so it must be escaped as `\a`).

### [I18N-R1-003] CRITICAL includes/connexion.php:20 -- Database charset set to 'utf8' instead of 'utf8mb4'

```php
mysqli_set_charset($base, 'utf8');
```

MySQL's `utf8` charset only supports up to 3-byte characters (Basic Multilingual Plane). It cannot store 4-byte Unicode characters including emoji, mathematical symbols, and some CJK characters. The BBCode system already converts smileys to emoji (e.g., `:D` becomes the emoji character U+1F603), and these emoji will be silently truncated or cause insertion errors when stored in the database. The `utf8mb4` charset supports the full Unicode range.

**Fix:** Change to `mysqli_set_charset($base, 'utf8mb4');` and ensure database tables/columns use `utf8mb4` collation.

## HIGH Findings

### [I18N-R1-004] HIGH includes/basicprivatephp.php:259 -- Email subject with accented characters not RFC 2047 encoded

```php
$sujet = "Debut d'une nouvelle partie";
```

The email subject contains the accented character `e` (in "Debut") sent as raw UTF-8. Per RFC 2047, non-ASCII characters in email headers must be encoded using `=?UTF-8?B?...?=` (Base64) or `=?UTF-8?Q?...?=` (Quoted-Printable). Many mail servers and clients will display the subject correctly due to UTF-8 auto-detection, but strict implementations may show garbled text.

**Fix:** Use `mb_encode_mimeheader($sujet, 'UTF-8', 'B')` or `=?UTF-8?B?` . base64_encode($sujet) . `?=`.

### [I18N-R1-009] HIGH includes/bbcode.php:317 -- htmlentities() used instead of htmlspecialchars()

```php
$text = htmlentities($text, ENT_QUOTES, 'UTF-8');
```

`htmlentities()` converts ALL characters that have HTML entity equivalents, including accented French characters: `e` becomes `&eacute;`, `a` becomes `&agrave;`, etc. This means that all forum posts, alliance descriptions, and any BBCode-processed text will have their French characters converted to HTML entities. While this renders correctly in browsers, it:
- Increases payload size unnecessarily
- Makes text less readable in page source
- Creates inconsistency with the rest of the codebase which uses `htmlspecialchars()` (only encodes `<`, `>`, `&`, `"`, `'`)
- Can cause issues if the text is later processed by code expecting literal UTF-8

**Fix:** Replace `htmlentities()` with `htmlspecialchars()` on this line.

### [I18N-R1-013] MEDIUM -- ucfirst() used on potentially accented strings (multiple files)

`ucfirst()` is not multibyte-safe. If a string begins with a multibyte UTF-8 character (e.g., an accented letter like `e`), `ucfirst()` will corrupt the first byte of the character rather than capitalizing it.

Affected locations:
- `includes/display.php:11` -- `ucfirst($nomsAccents[$num])` (resource names may have accents)
- `includes/basicprivatehtml.php:350` -- `ucfirst($nomsAccents[$num])`
- `marche.php:423` -- `ucfirst($nomsAccents[$num])`
- `marche.php:457,478,573` -- `ucfirst($ressource)` (resource names are ASCII, safe)
- `ecriremessage.php:10` -- `ucfirst(trim($_POST['destinataire']))` (player names)
- `allianceadmin.php:77,180,304` -- `ucfirst(trim(...))` on player names
- `includes/player.php:526` -- `ucfirst($nom)` (building names, ASCII)

Currently, the `$nomsAccents` array values are actually unaccented (`hydrogene` not `hydrogene`), so the bug is latent. But if accents are ever added to these arrays (as they should be -- see I18N-R1-027), `ucfirst()` will corrupt them.

**Fix:** Use `mb_convert_case($string, MB_CASE_TITLE, 'UTF-8')` or `mb_strtoupper(mb_substr($string, 0, 1, 'UTF-8'), 'UTF-8') . mb_substr($string, 1, null, 'UTF-8')` for the first character.

### [I18N-R1-018] MEDIUM includes/display.php:42 -- number_format uses non-breaking space for thousands separator

```php

```

The `separerZeros()` function uses regular space `' '` as both the decimal separator and thousands separator. In French formatting, the thousands separator should be a thin non-breaking space (U+202F) or non-breaking space (U+00A0) to prevent line breaks in the middle of numbers. The current regular space can cause numbers to wrap across lines.

This function is called throughout the codebase via `number_format($x, 0, ' ', ' ')` directly in many files (marche.php, classement.php, alliance.php, armee.php).

**Fix:** Use `number_format($nombre, 0, ',', "\xC2\xA0")` for proper French formatting with non-breaking space, or at minimum document the choice.

### [I18N-R1-045] LOW -- No Content-Security-Policy or X-Content-Type-Options headers

While not strictly i18n, the lack of `X-Content-Type-Options: nosniff` header means browsers may MIME-sniff responses and potentially misinterpret the charset. Combined with the inconsistent charset declarations (I18N-R1-001), this could lead to encoding issues.

**Fix:** Add `X-Content-Type-Options: nosniff` and ensure `Content-Type: text/html; charset=utf-8` is sent for all HTML responses.
