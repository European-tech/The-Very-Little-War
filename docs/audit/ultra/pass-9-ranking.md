## RANK-P9-001

**ID:** RANK-P9-001
**Severity:** MEDIUM
**File:** classement.php:35-56
**Title:** Player search POST form has no CSRF token

**Description:**
The player search form (sub=0, logged-in branch) posts `joueurRecherche` to `classement.php` via a `<form>` generated by the `item()` helper. The form action is `classement.php?sub=0`. There is no call to `csrfField()` or `csrfCheck()` anywhere in `classement.php`. A CSRF attack can force an authenticated player's browser to submit a search for a chosen player name; the response will reveal that player's rank page (through `$pageParDefaut`). While the direct damage is low (the attacker learns nothing they cannot learn by visiting the leaderboard themselves), the absence of CSRF protection is a policy violation and sets a bad precedent.

**Impact:** Low-severity CSRF (information disclosure of rank). The game's stated policy requires CSRF tokens on all POST forms.

**Recommended fix:** Add `<?= csrfField() ?>` inside the search form and call `csrfCheck()` at the top of the POST-processing block (lines 35-56).

## RANK-P9-002

**ID:** RANK-P9-002
**Severity:** LOW
**File:** classement.php:373
**Title:** `recalculerStatsAlliances()` called on every public GET of the alliance tab — unauthenticated write amplification

**Description:**
When `?sub=1` is requested and the game is not in maintenance mode, `recalculerStatsAlliances()` is called unconditionally regardless of whether the requester is authenticated. `recalculerStatsAlliances()` is a full O(A×M) write workload: it iterates every alliance, issues a `SELECT … FOR UPDATE` over all members, then writes an `UPDATE alliances SET …` for each. Any anonymous visitor (the page falls through to `basicpublicphp.php`, which does not gate access) can trigger this work repeatedly.

**Specifics verified:**
- `classement.php` line 3-10: no session required; anonymous users reach line 373.
- `player.php:866-891`: confirms the function issues `FOR UPDATE` locks inside a transaction for every alliance on every call.
- Multiple prior audit passes (PERF-R1-013, PERF-R2-012, P2-D6-007) identified this issue. It was not fixed.

**Impact:** A bot hammering `classement.php?sub=1` will induce heavy write contention on the `alliances` table. This is a DoS vector compounded by no authentication requirement.

**Recommended fix:** Move `recalculerStatsAlliances()` to a cron job (every 5 minutes) or gate the call behind `isset($_SESSION['login'])` and a per-session throttle.

## RANK-P9-003

**ID:** RANK-P9-003
**Severity:** LOW
**File:** classement.php:100-108
**Title:** Daily leaderboard query missing index on `derniereConnexion` join condition

**Description:**
The daily leaderboard query (line 100-108) filters on `m.derniereConnexion >= ?` via a JOIN between `autre` (a) and `membre` (m). Migration `0002_fix_column_types.sql:114` adds `idx_membre_derniereConnexion` on `membre.derniereConnexion`. However, the `DENSE_RANK() OVER (ORDER BY a.<order> DESC)` window function must also sort on the `autre` column chosen by `$order`. There are no indexes on the non-default sort columns (`pointsAttaque`, `pointsDefense`, `ressourcesPillees`, `tradeVolume`, `victoires`, `points`, `batmax`). Each non-default sort requires a full scan of the joined result set, negating the benefit of the single `idx_totalPoints` index.

**Evidence:** Migration `0026_add_totalpoints_index.sql` only creates `idx_totalPoints`. No migration adds indexes on the other seven `$order` columns.

**Impact:** Page-load latency spike when users click non-default leaderboard sort columns. At small player counts this is negligible; it becomes a problem as the player base grows.

**Recommended fix:** Add a composite or individual index on each of `autre`'s sortable columns: `(pointsAttaque)`, `(pointsDefense)`, `(ressourcesPillees)`, `(tradeVolume)`, `(victoires)`, `(points)`, and `(batmax)` in a new migration.

## RANK-P9-004

**ID:** RANK-P9-004
**Severity:** INFO
**File:** classement.php:101-108
**Title:** Daily leaderboard silently caps at `LEADERBOARD_PAGE_SIZE` with no pagination

**Description:**
The daily mode query (lines 100-108) applies `LIMIT LEADERBOARD_PAGE_SIZE` (20 rows) and has no pagination controls. The variable `$pages` is set to an empty string (`''`) at line 166 for the daily mode, so pagination links are suppressed. If more than 20 players have logged in today, only the top 20 appear, with no way to see the rest and no indication that the list is truncated.

**Impact:** UX issue for mid-game where many players are active daily; not a security finding. Listed for completeness.

**Recommended fix:** Add pagination to the daily leaderboard or display a "showing top 20 active players today" notice to set accurate expectations.

## RANK-P9-006

**ID:** RANK-P9-006
**Severity:** INFO
**File:** classement.php:16-33, 400-413, 640-648
**Title:** Column whitelisting confirmed correct for all three ranking tabs

**Description:**
Three separate `$order` whitelists exist in `classement.php`:
- Player tab (lines 16-33): maps `$clas` integers 0-6 to hardcoded column names (`batmax`, `victoires`, `pointsAttaque`, `pointsDefense`, `ressourcesPillees`, `points`, `tradeVolume`, default `totalPoints`).
- Alliance tab (lines 400-413): maps `$clas` to `totalConstructions`, `totalAttaque`, `totalDefense`, `totalPillage`, `pointsVictoire`, default `pointstotaux`.
- Forum tab (lines 640-648): maps `$clas` to `bombe`/`autre`, `troll`/`membre`, default `nbMessages`/`autre`.

All assignments use `elseif ($clas === N)` strict integer comparisons (MED-046 already remediated). No user-controlled string ever reaches the SQL ORDER BY clause. SQL injection via the sort parameter is not possible.

**Impact:** No vulnerability. Confirmed safe.

**Recommended fix:** None.

## RANK-P9-007

**ID:** RANK-P9-007
**Severity:** INFO
**File:** classement.php:151, 294, 483, 693; includes/player.php:855-864; includes/db_helpers.php:52-56
**Title:** XSS in username and alliance tag display — confirmed properly escaped

**Description:**
Player usernames are rendered via `joueur($donnees['login'])` (player.php:855-864). This function calls `htmlspecialchars($joueur, ENT_QUOTES, 'UTF-8')` before placing the value in both the `href` attribute and the visible text. Alliance tags are rendered via `alliance($donnees['tag'])` (db_helpers.php:52-56) which applies the same `htmlspecialchars` treatment.

Numeric fields displayed directly (`totalPoints`, `points`, `pointsAttaque`, etc.) pass through `number_format()` or `(int)` casts, which cannot produce XSS.

The prestige PP column (lines 158, 303) reads from an integer cache and uses a ternary that outputs `0` on miss — safe.

**Impact:** No XSS vulnerability found.

**Recommended fix:** None.

## RANK-P9-008

**ID:** RANK-P9-008
**Severity:** INFO
**File:** classement.php:102, 208
**Title:** DENSE_RANK usage confirmed correct for both player and daily leaderboards

**Description:**
Both the main player leaderboard (line 208) and the daily leaderboard (line 102) use `DENSE_RANK() OVER (ORDER BY a.<order> DESC)` as a window function in MariaDB. The rank computation for "find my page" also uses the correct dense-rank approximation: `COUNT(DISTINCT a.<order>) AS rank … WHERE a.<order> > ?` (lines 44, 179), which counts the number of distinct score values above the player's score and adds 1. This correctly emulates DENSE_RANK semantics for page calculation.

Alliance ranking (line 416) similarly uses `DENSE_RANK() OVER (ORDER BY <order> DESC)`.

**Impact:** No bug found. Ties are handled correctly.

**Recommended fix:** None.

## RANK-P9-009

**ID:** RANK-P9-009
**Severity:** INFO
**File:** classement.php:198-201, 393-396, 536-539, 616-619
**Title:** Pagination parameter validated on all four leaderboard tabs

**Description:**
All four leaderboard tabs (sub=0,1,2,3) validate `$_GET['page']` via `intval()` and clamp the result between 1 and `$nombreDePages`:
- `$page = isset($_GET['page']) ? intval($_GET['page']) : $pageParDefaut;`
- `if ($page < 1 || $page > $nombreDePages) { $page = $pageParDefaut; }`

The page value is used only in `LIMIT ?, ?` with integer bind parameters (`'ii'`), so no SQL injection is possible via the page parameter.

**Impact:** No pagination injection or out-of-bounds page access possible.

**Recommended fix:** None.

## RANK-P9-010

**ID:** RANK-P9-010
**Severity:** INFO
**File:** classement.php:69-86, 353-374
**Title:** Frozen rankings during season maintenance — confirmed implemented

**Description:**
Both the player ranking tab (sub=0, lines 69-86) and the alliance ranking tab (sub=1, lines 353-374) check the maintenance flag before executing any ranking query. When `maintenance == 1`, both tabs display a "Classement gelé — fin de saison en cours" banner and skip all live queries, including the write-path `recalculerStatsAlliances()`. The maintenance check uses two fallback paths: first from the `$maintenance` variable populated by `basicprivatephp.php` for authenticated users, then from a direct DB query for public/unauthenticated visitors.

**Impact:** Rankings are correctly frozen during season reset. No data race between live queries and the wipe.

**Recommended fix:** None.

## RANK-P9-011

**ID:** RANK-P9-011
**Severity:** INFO
**File:** includes/formulas.php:105-135
**Title:** Sqrt ranking formula correctly implemented and capped

**Description:**
`calculerTotalPoints()` (formulas.php:105-116) implements:
```
totalPoints = round(
    RANKING_CONSTRUCTION_WEIGHT * pow(max(0, construction), 0.5)
  + RANKING_ATTACK_WEIGHT      * pow(max(0, attaque),       0.5)
  + RANKING_DEFENSE_WEIGHT     * pow(max(0, defense),       0.5)
  + RANKING_TRADE_WEIGHT       * pow(max(0, cappedCommerce), 0.5)
  + RANKING_PILLAGE_WEIGHT     * pow(max(0, pillage),        0.5)
)
```
Weights (1.0/1.5/1.5/1.0/1.2) and exponent (0.5) are defined as named constants in `config.php:497-502`. The trade volume input is capped at `TRADE_VOLUME_CAP` (10M) before the sqrt is applied, preventing trade-volume inflation (MEDIUM-016). The `max(0, …)` guards prevent `pow()` on negative inputs. `recalculerTotalPointsJoueur()` (lines 122-135) recomputes `totalPoints` from raw columns after any score change. No integer overflow risk for the expected score ranges.

**Impact:** Formula is mathematically correct and manipulation-resistant.

**Recommended fix:** None.
